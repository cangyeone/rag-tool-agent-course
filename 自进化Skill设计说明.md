# 自进化 Skill 怎么做

## 1. 基本理解

Skill 可以理解成“给智能体看的工作说明书”。普通 Skill 是固定说明书：遇到某类任务，就按里面的流程做。自进化 Skill 不是让它偷偷改自己，而是让它在每次任务后把问题、失败原因、好用做法记录下来，再由人确认后更新说明书。

更稳的说法是：**可迭代 Skill**，不要说成“完全自动进化”。业务系统里涉及客服口径、安全判断、工单流程和权限边界，不能让模型自己随便改规则。

## 2. 自进化 Skill 的核心闭环

```text
课堂/业务任务
  -> 调用 Skill
  -> 执行脚本或流程
  -> 记录结果和失败点
  -> 生成改进建议
  -> 人工确认
  -> 更新 Skill / 参考资料 / 测试集
  -> 下次任务复用
```

这条链路的重点不是“模型变聪明了”，而是经验被沉淀下来了。下一次遇到类似问题，不用重新解释一遍。

## 3. 一个 Skill 文件夹应该长什么样

```text
railway-rag-assistant/
├── SKILL.md
├── references/
│   ├── railway_rules.md
│   ├── common_failures.md
│   └── test_cases.md
├── scripts/
│   ├── build_index.py
│   ├── run_eval.py
│   └── update_suggestion.py
└── assets/
    └── input_template.txt
```

建议分工：

- `SKILL.md`：只写最核心的流程和触发条件，不要写太长。
- `references/`：放行业规则、常见问题、失败案例、测试题。
- `scripts/`：放稳定可重复执行的代码，比如构建索引、跑评测、生成改进建议。
- `assets/`：放模板、示例输入、课程素材。

## 4. SKILL.md 写什么

一个行业 RAG 助手 Skill 可以这样写：

```markdown
---
name: railway-rag-assistant
description: 当用户需要构建、测试、优化行业 RAG 问答助手，或处理 示例业务系统 客服、安全制度、巡检记录、工单知识库时使用。
---

# Railway RAG Assistant

## 工作流程

1. 先确认任务属于哪类：客服问答、安全制度、巡检记录、运维工单、培训答疑。
2. 查 `references/railway_rules.md`，确认能不能回答。
3. 如果需要实时状态，优先使用工具或接口，不让模型凭空猜。
4. 生成回答时必须保留业务边界：不承诺真实库存，不替代现场安全判断。
5. 回答后记录是否命中资料、是否调用工具、是否出现失败。

## 需要更新 Skill 时

只有在用户明确要求，或评测发现同类错误重复出现时，才提出更新建议。
更新前先写清楚：
- 遇到了什么问题
- 原 Skill 哪里不够
- 准备改哪一段
- 用哪些测试问题验证
```

## 5. 自进化的“记忆”放在哪里

不要把所有运行记录都塞进 `SKILL.md`。更好的做法是分三类放：

| 内容 | 放哪里 | 例子 |
|---|---|---|
| 稳定流程 | `SKILL.md` | “先检索资料，再调用工具，再生成回答” |
| 常见失败 | `references/common_failures.md` | “模型承诺候补申请一定成功” |
| 测试问题 | `references/test_cases.md` | “身份证忘带还能进站吗？” |
| 可执行检查 | `scripts/run_eval.py` | 批量跑测试题，统计是否越界 |

这样 Skill 不会越来越臃肿，也便于演示讲“为什么要分层沉淀经验”。

## 6. 自进化不能全自动

行业场景里建议加三道闸：

1. **只记录，不自动修改**  
   每次失败先写入日志或改进建议，不直接改 Skill。

2. **人工确认后再改**  
   涉及规则口径、安全判断、权限、接口调用，都要人工确认。

3. **改完必须回归测试**  
   用固定测试集检查：候补申请、退款、学生优惠、证件、安全检查、工单查询等问题是否还稳定。

## 7. 课堂上可以怎么演示

可以用一个很小的例子：

### 第一步：原始 Skill 不完整

用户问：

```text
候补申请是不是一定能成功？
```

模型错误回答：

```text
一般可以成功，请耐心等待。
```

问题：它没有说明“不能保证成功”。

### 第二步：记录失败

```json
{
  "case": "候补申请是不是一定能成功？",
  "failure": "回答过度承诺",
  "expected_rule": "候补申请是否成功取决于库存变化，不能承诺一定成功",
  "suggestion": "在 Skill 的边界要求中加入候补申请成功率提醒"
}
```

### 第三步：人工确认后更新 Skill

在 `SKILL.md` 或 `references/common_failures.md` 中加入：

```markdown
候补申请类问题必须提醒：
- 候补申请提交不等于下单成功
- 是否兑现取决于后续库存变化
- 以 官方页面显示为准
```

### 第四步：再跑测试

固定测试题：

```text
1. 候补申请一定能成功吗？
2. G107 没票了还能怎么办？
3. 候补申请截止时间到了还没兑现怎么办？
```

如果三条都能稳定回答，并且不再承诺“一定成功”，这次 Skill 更新才算有效。

## 8. 和 RAG、Agent、可视化工具 的关系

| 模块 | 在自进化里负责什么 |
|---|---|
| RAG | 提供规则、案例、失败记录和测试题 |
| Agent | 执行“检索、调用工具、生成、质检、记录失败”这条链路 |
| 可视化工作流 | 把这条链路做成可视化节点，便于演示演示和交付 |
| Skill | 沉淀做事方法，让下一次任务不用从零开始 |

## 9. 课堂结论

自进化 Skill 不是让模型自己长出新规则，而是把每次任务里的有效经验沉淀下来。行业更适合“受控迭代”：失败可记录，建议可生成，更新要确认，结果要测试。这样既能越用越顺，又不会让模型越权。
