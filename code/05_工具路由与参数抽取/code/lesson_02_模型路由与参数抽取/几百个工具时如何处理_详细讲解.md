# 几百个工具时如何处理：规则过滤、向量召回、LLM rerank、schema 注入

Agent 系统里的工具数量一多，难点就不再只是“怎么写一个 tool schema”，而是“怎么从很多工具里选出这一轮真正需要的几个”。

如果系统有 300 个工具，每个工具都有完整参数说明、字段解释、枚举值和示例，把它们一次性放进模型上下文，通常不是好办法。上下文会很长，模型也会被大量无关工具干扰。更稳的做法是先筛工具，再给 schema。

## 1. 总体链路

推荐链路如下：

```text
用户问题
  ↓
规则过滤：根据关键词、业务域、权限、当前页面等强信号锁定工具组
  ↓
向量检索：从候选工具 description 中召回 top-k
  ↓
LLM rerank：让模型判断工具是否真的相关、是否需要组合
  ↓
schema 注入：只给最终 3-10 个工具的完整参数定义
  ↓
正式 tool call：模型选择工具并生成 arguments
```

这条链路的目标很明确：不要让模型在几百个完整工具 schema 里盲选。

## 2. 第一步：规则过滤

规则过滤抓的是明显、稳定、低成本的信号。

例子：

| 用户表达 | 命中工具组 | 说明 |
| --- | --- | --- |
| “帮我找一下邮件” | Gmail tool group | 邮件意图很明确 |
| “生成一个 演示材料” | presentation tool group | 演示材料 / 演示材料 / 课程材料都是强信号 |
| “代码仓库里搜一下” | filesystem / code tools | 代码仓库、repo、文件系统都是强信号 |
| “查知识库里的条款” | RAG tools | 知识库、文档、检索是强信号 |
| “建一个日程” | Calendar tools | 日程、会议、提醒是强信号 |

规则过滤不是为了最终决定工具，而是先缩小范围。

比如系统有 300 个工具，命中 “演示材料” 后，可能只剩 12 个 演示材料工具。后面的向量检索和模型判断就轻得多。

### 规则过滤适合放哪些信号

- 用户明确关键词：邮件、演示材料、表格、代码仓库、数据库、知识库
- 当前产品页面：用户在邮件页面提问，优先邮件工具
- 用户权限：没有写权限，就不要进入写文件工具组
- 租户配置：客户没有启用 Gmail，就过滤 Gmail 工具
- 风险级别：涉及删除、转账、发邮件等动作，先进入安全检查

### 规则过滤不要做得太死

规则过滤容易漏召回。比如用户说“帮我把会议材料整理成几页”，没有直接说 演示材料，但可能需要 演示材料工具。

所以规则过滤最好有兜底：

- 命中强规则：进入对应工具组
- 没命中规则：进入全工具池或常用工具池做向量检索
- 命中多个规则：保留多个工具组，不要过早丢掉

## 3. 第二步：向量检索

规则过滤之后，候选工具仍然可能不少。向量检索用来做语义召回。

向量检索的对象不是完整 schema，而是轻量 tool card：

```json
{
  "name": "presentation_create_doc",
  "group": "presentation",
  "description": "创建 演示材料 或 Google Slides 演示材料，生成课程、汇报、培训课程材料。",
  "short_schema": {
    "title": "string",
    "outline": "array"
  }
}
```

这类 tool card 很短，适合建索引。

### 为什么不用完整 schema 做向量检索

完整 schema 里有大量参数字段，很多字段对语义检索是噪声。

比如一个 演示材料 工具的完整 schema 可能包含：

- title
- page_count
- theme
- layout
- image_policy
- speaker_notes
- export_format
- language

用户只是说“帮我做 演示材料”，这些字段不用在召回阶段全部参与。

### 向量检索适合解决什么问题

规则过滤依赖关键词，向量检索能处理表达变化。

例子：

| 用户说法 | 可能召回 |
| --- | --- |
| “做一份演示材料” | presentation_create_doc |
| “把这段内容整理成课程材料” | presentation_create_doc |
| “查一下项目里哪里用了这个函数” | repo_search_code |
| “找上周客户发来的资料” | gmail_search_email |

向量检索的输出一般是 top-k，比如 top-10、top-20。

## 4. 第三步：LLM rerank

向量检索负责“召回像的”，LLM rerank 负责“判断能不能用”。

rerank 阶段可以让小模型或主模型看这些内容：

```text
用户问题：
帮我找一下张三上周发来的邮件，看看附件里有没有培训课程材料。

候选工具：
1. gmail_search_email：搜索 Gmail 邮件，按发件人、主题、正文关键词、时间范围查询邮件。
2. gmail_send_email：发送 Gmail 邮件。
3. presentation_create_doc：创建 演示材料。
4. file_search：搜索本地文件。

请判断每个工具是否相关，输出排序和理由。
```

rerank 输出可以是结构化 JSON：

```json
[
  {
    "tool_name": "gmail_search_email",
    "keep": true,
    "score": 95,
    "reason": "用户要找邮件和附件，搜索邮件是第一步"
  },
  {
    "tool_name": "presentation_create_doc",
    "keep": false,
    "score": 30,
    "reason": "用户提到课程材料，但当前动作是查找邮件，不是创建 演示材料"
  }
]
```

### rerank 的价值

向量检索可能把 “演示材料 工具”召回来，因为用户说了“培训课程材料”。但这句话里的“培训课程材料”是邮件附件，不是要新建 演示材料。

LLM rerank 能结合句子意图判断：

- 当前要找邮件，不是创建 演示材料
- Gmail 搜索应该排在前面
- 演示材料 工具可以不注入完整 schema

这一步能明显减少误调用。

## 5. 第四步：schema 注入

schema 注入是最后一步。

只有最终留下来的 3-10 个工具，才给完整 schema。

完整 schema 包括：

- 工具名称
- 工具说明
- 参数字段
- required 字段
- enum 选项
- 参数解释
- 调用限制

比如最后只注入：

```json
[
  {
    "type": "function",
    "function": {
      "name": "gmail_search_email",
      "description": "搜索 Gmail 邮件，适合查找某封邮件、会议通知、附件邮件或客户来信。",
      "parameters": {
        "type": "object",
        "properties": {
          "query": {"type": "string", "description": "邮件检索关键词"},
          "from_user": {"type": "string", "description": "可选，发件人邮箱或姓名"},
          "date_range": {"type": "string", "description": "可选，例如 last_7_days"},
          "has_attachment": {"type": "boolean", "description": "是否只查带附件邮件"}
        },
        "required": ["query"]
      }
    }
  }
]
```

模型正式选择工具时，只在这些工具里选。

## 6. 为什么最终是 3-10 个工具

不是绝对数字，但工程上很常见。

少于 3 个，可能漏掉组合工具。

多于 10 个，模型选择成本变高，误选概率增加。

一般可以这样取：

| 任务类型 | 建议最终工具数 |
| --- | --- |
| 简单问答 | 1-3 个 |
| 文件/邮件/日程类操作 | 3-5 个 |
| 复杂 Agent 任务 | 5-10 个 |
| 高风险操作 | 少量工具 + 明确确认步骤 |

## 7. 一个完整例子

用户说：

```text
帮我找一下张三上周发来的邮件，看看附件里有没有培训课程材料。
```

### 规则过滤

命中：

```text
邮件 → Gmail tool group
```

候选从 300 个缩到 Gmail 组的 8 个。

### 向量检索

从 Gmail 组里召回：

1. gmail_search_email
2. gmail_list_attachments
3. gmail_download_attachment
4. gmail_summarize_email

### LLM rerank

模型判断：

1. gmail_search_email：必须保留
2. gmail_list_attachments：可能需要
3. gmail_download_attachment：可能需要
4. gmail_send_email：不需要

### schema 注入

最终注入：

- gmail_search_email
- gmail_list_attachments
- gmail_download_attachment

模型正式生成 tool call：

```json
{
  "name": "gmail_search_email",
  "arguments": {
    "query": "培训课程材料",
    "from_user": "张三",
    "date_range": "last_7_days",
    "has_attachment": true
  }
}
```

## 8. 落地时的缓存设计

几百个工具的场景，工具检索不要每轮都从零开始。

可以缓存：

- tool description embedding
- group → tools 映射
- 用户最近使用的工具
- 当前应用页面对应的工具组
- 权限过滤后的工具列表
- rerank 结果短期缓存

用户连续问同一类问题时，可以优先使用最近工具组。

## 9. 权限和安全

工具筛选不只是相关性问题，还要考虑权限。

例如：

- 用户没有邮箱权限，Gmail 工具不能进入候选
- 用户只能读代码，不能注入 repo_edit_file
- 删除文件、发送邮件、提交工单需要二次确认
- 高风险工具不能只靠模型判断

推荐顺序：

```text
权限过滤
  ↓
规则过滤
  ↓
向量检索
  ↓
LLM rerank
  ↓
风险检查
  ↓
schema 注入
```

## 10. 和 工作流平台 / Agent 框架的关系

工作流平台、LangGraph、AutoGen、OpenAI Agents SDK、Claude tool use、Qwen Function Calling，本质上都会遇到同一个问题：

```text
工具太多时，不能把全部工具一次性给模型。
```

平台可能帮你做一部分工具管理，但业务系统仍然需要设计：

- 工具分组
- 工具描述
- 工具权限
- 工具召回
- 工具 rerank
- 工具调用日志
- 工具失败兜底

## 11. 课堂讲解建议

先用一句话概括：

```text
几百个工具时，不是让模型从几百个工具里盲选，而是先帮模型缩小选择范围。
```

再按四步展开：

1. 规则过滤：快，抓强信号
2. 向量检索：广，抓语义相似
3. LLM rerank：准，判断真实相关性
4. schema 注入：省，只给少量完整参数定义

最后用 `07_几百个工具_端到端调用链路.py` 跑四个例子：

- 邮件
- 演示材料
- 代码仓库
- 知识库

让学员观察每一步候选工具数量如何下降。
