# 多轮对话：其他开源模型与国际模型的常见做法

这份文档用于补充课堂代码，说明不同模型平台在多轮对话上的实现方式。不同厂商的字段名称不完全一样，但底层逻辑很接近：模型本身通常不替业务系统保存完整会话，应用程序需要管理历史消息、摘要、工具结果和上下文预算。

## 1. 先把概念说清楚

多轮对话不是模型“真的记住了用户”。在大多数 API 里，每次请求都是一次新的 HTTP 请求。模型能接上上一轮，是因为程序把历史对话重新传给了模型。

常见做法有三种：

| 做法 | 说明 | 适合场景 |
| --- | --- | --- |
| 每次传完整 `messages` | 程序维护所有历史消息，每轮请求都带上 | 最通用，DeepSeek、Claude、Qwen、开源模型都适用 |
| 传上一轮响应 ID | 服务端保存一段响应链，下一轮只传 `previous_response_id` | OpenAI Responses API 等支持服务端状态的接口 |
| 使用会话对象 | 服务端维护一个 conversation/session，后续请求绑定该会话 | 平台型应用、Agent 服务、托管工作流 |

工程里一般不会无限传完整历史。真实系统会做历史裁剪、摘要压缩、重要事实提取、RAG 补充和工具结果筛选。

## 2. OpenAI / GPT-5.5 的做法

OpenAI 当前推荐使用 Responses API 管理更复杂的对话、工具调用和状态。GPT-5.5 属于 OpenAI GPT-5 系列的新模型，可通过 Responses API 和官方 SDK 调用。官方模型页说明，GPT-5.5 面向复杂推理、编码和专业任务，支持文本与图像输入、文本输出；模型指南也强调需要根据新模型重新调试 reasoning effort、verbosity、工具说明和输出格式。

OpenAI 多轮对话主要有三种写法。

### 2.1 手动传历史

这是最容易理解的写法：业务系统自己维护历史，每次请求都把历史消息传进去。

```python
from openai import OpenAI

client = OpenAI()

messages = [
    {"role": "developer", "content": "回答要简洁、可靠。"},
    {"role": "user", "content": "什么是多轮对话？"},
]

response = client.responses.create(
    model="gpt-5.5",
    input=messages,
)

answer = response.output_text
messages.append({"role": "assistant", "content": answer})
messages.append({"role": "user", "content": "它和单轮问答有什么区别？"})

response = client.responses.create(
    model="gpt-5.5",
    input=messages,
)

print(response.output_text)
```

这类写法的好处是透明，所有上下文都在应用代码里。缺点是历史越长，请求越大，费用和延迟都会增加。

### 2.2 使用 `previous_response_id`

Responses API 支持用上一轮 response 的 ID 串起下一轮。应用程序不用每次手动拼完整消息，但仍要理解：服务端会根据响应链恢复上下文。

```python
from openai import OpenAI

client = OpenAI()

first = client.responses.create(
    model="gpt-5.5",
    input="用两句话解释什么是 RAG。",
)

second = client.responses.create(
    model="gpt-5.5",
    previous_response_id=first.id,
    input="刚才说的检索，具体检索的是什么？",
)

print(second.output_text)
```

这种写法更简洁，适合快速搭建连续对话。工程上仍然要注意日志、审计、历史保留周期、成本和上下文长度。

### 2.3 使用 conversation / 会话状态

OpenAI 文档还提供 conversation state 的写法。可以把多轮交互绑定到同一个 conversation，适合更长的会话、工具调用和复杂应用。

课堂里可以这样理解：

- `messages`：自己背着全部历史去问模型。
- `previous_response_id`：拿上一轮的编号，让平台接着上一轮答。
- `conversation`：把多轮对话放进一个平台会话里管理。

### 2.4 GPT-5.5 的 reasoning 内容要不要传回？

对 OpenAI Responses API 来说，通常不需要把模型内部 reasoning 当成普通历史消息手动拼回去。业务系统更应该保存这些内容：

- 用户问题
- 模型最终回答
- 工具调用记录
- 工具返回结果
- 引用来源
- 关键业务状态

如果使用 `previous_response_id` 或 conversation state，平台会按接口语义处理上一轮状态。不要把内部推理文本当成普通 `assistant` 回答随意塞回历史，否则容易增加噪声，也可能让后续回答变得不稳定。

## 3. Claude / Anthropic 的做法

Claude Messages API 是典型的无状态多轮对话。Anthropic 官方文档明确说明：多轮时需要把完整会话历史发送给 API。

Claude 的消息结构通常是：

```python
import anthropic

client = anthropic.Anthropic()

message = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system="你是一个课堂助手，回答要简洁。",
    messages=[
        {"role": "user", "content": "什么是多轮对话？"},
        {"role": "assistant", "content": "多轮对话是模型结合历史上下文回答后续问题。"},
        {"role": "user", "content": "那上下文是谁保存的？"},
    ],
)

print(message.content[0].text)
```

Claude 的特点：

- `system` 通常是顶层字段。
- `messages` 里主要是 `user` 和 `assistant`。
- API 无状态，历史由应用程序维护。
- 工具调用、多模态、长上下文都仍然要考虑上下文预算。

Claude 的 extended thinking 和工具调用也有专门规则。课堂里不建议把“思考过程”当成普通聊天历史反复塞回去，除非文档明确要求某个字段必须延续。

## 4. Google Gemini 的做法

Gemini API 使用 `contents` 表示多轮内容。官方 API 说明里，多轮对话需要传入多个 `Content` 对象，角色通常在 `user` 和 `model` 之间交替。SDK 的 chat 接口可以帮你管理历史；如果直接用 REST API，就要自己维护历史。

简化写法如下：

```python
from google import genai

client = genai.Client()

chat = client.chats.create(model="gemini-3-pro")

response1 = chat.send_message("请解释什么是多轮对话。")
print(response1.text)

response2 = chat.send_message("那上下文是怎么传递的？")
print(response2.text)
```

如果不用 SDK 的 chat 包装，而是直接调用底层接口，就需要自己构造类似这样的历史：

```python
contents = [
    {"role": "user", "parts": [{"text": "什么是多轮对话？"}]},
    {"role": "model", "parts": [{"text": "多轮对话会结合历史问题和回答。"}]},
    {"role": "user", "parts": [{"text": "那历史是谁保存的？"}]},
]
```

Gemini 的多模态输入也放在 `parts` 里。文字、图片、音频、视频等内容都可以成为一轮输入的一部分。

## 5. DeepSeek 的做法

DeepSeek 的普通多轮和工具调用规则已经在代码里单独演示。这里再做一个对照总结。

| 场景 | 做法 |
| --- | --- |
| 普通多轮 | 每轮把历史 `messages` 传回；不发生 tool call 时，`reasoning_content` 不需要传回 |
| thinking + tool calls | 带 `tool_calls` 的 assistant 消息要完整保留，包括 `reasoning_content` |
| OpenAI SDK 调 DeepSeek | 使用 `OpenAI(api_key=..., base_url="https://api.deepseek.com")` |

DeepSeek 的特殊点是：thinking 模式下如果发生 tool call，后续请求需要完整传回那轮 assistant 的 `reasoning_content`。这和普通多轮不同。

## 6. 开源模型的本地多轮对话

开源模型本地运行时，常见方式有三类。

### 7.1 Transformers + chat template

Hugging Face Transformers 的核心做法是：

1. 用 `messages` 保存历史。
2. 用 tokenizer 或 processor 的 `apply_chat_template` 把历史渲染成模型输入。
3. 调用 `generate()`。
4. 把模型输出追加到 `messages`。

典型代码：

```python
messages = [
    {"role": "system", "content": "你是一个课堂助手。"},
    {"role": "user", "content": "什么是多轮对话？"},
]

prompt = tokenizer.apply_chat_template(
    messages,
    tokenize=False,
    add_generation_prompt=True,
)

inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
outputs = model.generate(**inputs, max_new_tokens=200)
answer = tokenizer.decode(outputs[0][inputs["input_ids"].shape[1]:], skip_special_tokens=True)

messages.append({"role": "assistant", "content": answer})
```

本地模型没有 API 服务器帮助保存历史。所有历史都在 Python 程序里。

### 7.2 vLLM / SGLang / LM Studio 的 OpenAI 兼容接口

很多本地推理服务会提供 OpenAI 兼容接口。例如 vLLM 的 OpenAI-compatible server 支持 Chat Completions API，调用方式和云端 OpenAI / DeepSeek 类似。

```python
from openai import OpenAI

client = OpenAI(
    api_key="EMPTY",
    base_url="http://localhost:8000/v1",
)

messages = [
    {"role": "system", "content": "你是一个课堂助手。"},
    {"role": "user", "content": "解释什么是多轮对话。"},
]

response = client.chat.completions.create(
    model="Qwen/Qwen3.5-0.8B",
    messages=messages,
)

answer = response.choices[0].message.content
```

注意：OpenAI 兼容接口只是“请求格式相似”。是否支持工具调用、thinking、JSON mode、多模态，要看具体服务和模型。

### 7.3 Ollama / llama.cpp

Ollama、llama.cpp 一类本地服务通常也支持 chat 风格输入。基本原则仍然一样：

- 客户端维护历史消息。
- 服务端把消息渲染成模型 prompt。
- 上下文窗口有限，历史太长要裁剪或摘要。

示意结构：

```python
messages = [
    {"role": "system", "content": "你是一个课堂助手。"},
    {"role": "user", "content": "什么是多轮对话？"},
    {"role": "assistant", "content": "多轮对话会利用前面的上下文。"},
    {"role": "user", "content": "那上下文太长怎么办？"},
]
```

无论底层是 Qwen、Llama、Mistral、Yi、GLM 还是 DeepSeek 开源权重，多轮对话的关键都不是模型名字，而是：

- chat template 是否正确
- 历史消息是否按角色组织
- 上下文是否超过窗口
- 是否需要摘要压缩
- 工具结果是否按模型要求放回

## 8. 多轮对话里的“记忆”怎么做

课堂上不要把 memory 讲成玄学。工程里的 memory 通常分四层。

| 层级 | 保存内容 | 常见做法 |
| --- | --- | --- |
| 短期上下文 | 最近几轮对话 | 直接放进 `messages` |
| 摘要记忆 | 较早历史的压缩版 | 用模型总结为几条事实 |
| 用户画像 | 稳定偏好和身份信息 | 数据库字段，需用户授权和合规 |
| 外部知识 | 文档、规则、系统记录 | RAG 检索后放入本轮上下文 |

一个实用策略：

1. 最近 3 到 6 轮完整保留。
2. 更早的内容压缩成摘要。
3. 重要事实结构化保存。
4. 文档类知识不要塞进聊天历史，用 RAG 按需检索。
5. 工具返回结果只保留关键字段，避免把大段 JSON 一直带下去。

## 8. 思考内容要不要保存

不同平台规则不一样，不能一概而论。

| 平台 / 模型 | 普通多轮是否保存思考内容 | 工具调用时是否保存 |
| --- | --- | --- |
| OpenAI GPT-5.5 / Responses API | 通常不手动保存内部推理文本 | 按 Responses API 的 response / tool item 机制处理 |
| Claude | 一般保存最终回答和工具结果 | 按 Claude tool use / extended thinking 文档处理 |
| Gemini | 通常保存 user / model 可见内容 | 工具调用按 function call / function response 结构处理 |
| DeepSeek | 普通多轮不需要传回 `reasoning_content` | thinking + tool calls 时需要完整传回 |
| 本地开源模型 | 通常没有独立 reasoning 字段 | 如果输出 `<think>...</think>`，建议解析后只保留最终回答 |

业务系统更应该保存可审计内容：

- 用户输入
- 最终回答
- 检索到的资料
- 工具调用参数
- 工具返回结果
- 错误信息
- 人工确认记录

思考文本不一定适合长期保存。它可能很长、成本高，也可能包含不适合直接展示或复用的中间推理。

## 9. 课堂对比讲法

可以按这条线讲：

1. 本地 Qwen：自己维护 `messages`，自己渲染 chat template。
2. DeepSeek：云端 API 也无状态，普通多轮仍然要传历史。
3. DeepSeek 工具调用：一旦工具调用参与，消息结构要保留完整。
4. OpenAI GPT-5.5：Responses API 可以用 `previous_response_id` 或 conversation 管理状态。
5. Claude / Gemini：形式不同，但本质也是“显式管理历史或使用 SDK 帮你管理历史”。

最后落到一句话：

> 多轮对话不是把所有内容无限塞回模型，而是把“下一轮回答真正需要的上下文”组织好，再交给模型。
