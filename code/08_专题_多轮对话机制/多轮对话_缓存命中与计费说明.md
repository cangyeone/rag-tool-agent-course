# 多轮对话里的缓存、输出和计费

多轮对话看起来像是模型一直记得前面聊过什么。工程上不是这样。多数模型 API 每次请求都要重新收到一份上下文，模型再根据这份上下文继续回答。

所以一轮对话的成本，通常不是只看用户刚刚输入的那一句话，而是看这次请求里实际送进模型的全部内容。

## 1. 一次请求里通常有哪些 token

一次多轮对话请求里，常见 token 可以分成三类：

| 类型 | 来自哪里 | 例子 |
| --- | --- | --- |
| 输入 token | 本轮发给模型的内容 | system 说明、历史消息、用户新问题、RAG 片段、工具结果 |
| 缓存命中 token | 输入 token 中和之前请求前缀相同、可复用的部分 | 固定 system、长文档前缀、重复的历史消息 |
| 输出 token | 模型本轮生成的新内容 | 最终回答、JSON、工具调用参数、推理模型的可计费输出部分 |

可以把它理解成：

```text
本轮费用 ≈ 未缓存输入 + 缓存命中输入 + 输出
```

不同平台的价格表会分得更细，有的平台还会把 reasoning、tool call、image input 等单独计量。课堂里先记住一个原则：**输入越长、输出越长，成本越高；输出通常比输入更贵；缓存命中可以降低重复输入的成本。**

## 2. 什么是缓存命中

缓存命中指的是：这次请求的开头部分，和前面某次请求的开头部分完全一致，模型服务可以复用前面已经处理过的结果。

典型能命中的内容：

- 固定的 `system` 消息。
- 一段长期不变的背景材料。
- 多轮对话中前几轮完全相同的历史。
- RAG 中被稳定放在同一位置的资料前缀。

典型不容易命中的内容：

- 每次都变化的时间戳、随机编号、追踪 ID。
- 把用户新问题放在最前面，导致后面的固定内容整体后移。
- 每轮都重新打乱 RAG 片段顺序。
- 在固定提示前面插入不同的临时说明。

缓存一般看的是“前缀”。也就是说，越靠前、越稳定的内容越容易命中。前面一旦变了，后面即使有相同文本，也可能吃不到同样的缓存收益。

## 3. 缓存命中和未命中的区别

下面是一个简化例子。

第一轮请求：

```text
system: 你是一个知识库问答助手，回答要引用资料。
资料: A 文档的长内容……
user: 什么是向量检索？
```

第二轮请求：

```text
system: 你是一个知识库问答助手，回答要引用资料。
资料: A 文档的长内容……
user: overlap 有什么用？
```

如果平台支持前缀缓存，第二轮里 `system + 资料` 这部分有机会命中缓存。模型服务不需要从头完整计算这一大段内容，延迟和费用都可能下降。

再看一个不容易命中的写法：

```text
user: overlap 有什么用？
system: 你是一个知识库问答助手，回答要引用资料。
资料: A 文档的长内容……
```

本轮问题放在最前面，每轮都会变。后面的固定资料虽然文字没变，但它不再是稳定前缀，缓存效果会差很多。

## 4. 输出 token 为什么经常更贵

输入 token 是模型读取已有内容。输出 token 是模型一步一步生成新内容。

生成时，模型通常要反复做这些事：

1. 根据当前上下文预测下一个 token。
2. 把新 token 拼回上下文。
3. 再预测下一个 token。
4. 一直生成到停止条件。

输出不是一次性写完的，而是逐 token 生成。回答越长，模型推理步骤越多，延迟也越明显。

这也是为什么同样 1,000 个 token，输入和输出的价格常常不一样。很多平台会把输出 token 定得更贵，因为生成比读取更占推理资源。

## 5. 为什么多轮对话计费差距会很大

同一个用户问题，费用可能差很多，主要差在上下文组织方式。

### 情况 A：每轮只传必要历史

```text
system: 固定要求
summary: 前面已经确认用户要做个人知识库
user: 那向量库怎么选？
```

特点：

- 输入短。
- 输出可控。
- 成本低。
- 适合已经做过摘要的长对话。

### 情况 B：每轮传完整历史

```text
system: 固定要求
第 1 轮 user / assistant
第 2 轮 user / assistant
第 3 轮 user / assistant
……
第 30 轮 user / assistant
user: 那向量库怎么选？
```

特点：

- 输入越来越长。
- 每轮都重复传很多旧内容。
- 如果缓存命中好，成本能降一些；如果命中不好，费用会明显上升。
- 对长会话不太稳，容易把无关旧信息带回来。

### 情况 C：RAG 每轮塞入大量资料

```text
system: 固定要求
history: 最近几轮对话
retrieved_docs: 20 段检索资料
user: 那向量库怎么选？
```

特点：

- 检索资料会显著增加输入 token。
- Top K 太大时，模型读了很多不一定有用的内容。
- 资料顺序每轮变化时，缓存命中也会变差。
- 回答可能更长，输出成本也会上升。

### 情况 D：要求模型输出很长

```text
请给出 5000 字详细方案。
```

特点：

- 输入可能不长。
- 输出很长。
- 输出 token 本身就贵，整体费用容易上去。
- 如果只是课堂演示，可以先限制 `max_tokens`。

## 6. 缓存命中时，接口里通常怎么看

不同平台的字段名称不完全一样。有的平台会在 `usage` 里显示类似这些信息：

```json
{
  "usage": {
    "prompt_tokens": 3200,
    "completion_tokens": 480,
    "total_tokens": 3680,
    "prompt_tokens_details": {
      "cached_tokens": 2400
    }
  }
}
```

可以这样读：

- `prompt_tokens`：本轮输入一共多少 token。
- `cached_tokens`：输入里有多少 token 命中了缓存。
- `completion_tokens`：模型生成了多少 token。
- `total_tokens`：总 token 数。

注意：`prompt_tokens` 仍然会显示完整输入长度。缓存命中不是说这部分不存在了，而是说这部分可能按更低成本或更快路径处理。具体折扣和字段要以所用平台文档为准。

## 7. 多轮对话中如何降低费用

### 7.1 固定内容放前面

推荐顺序：

```text
system 固定要求
长期固定背景
少量稳定示例
历史摘要
检索资料
用户本轮问题
```

固定内容越靠前，越容易形成缓存前缀。

### 7.2 不要每轮都传完整历史

长对话可以保留三类信息：

- 最近几轮原文。
- 前面内容的短摘要。
- 已确认的重要事实。

旧对话不用全部塞回去。很多时候，一段 200 字摘要比 20 轮历史更可靠。

### 7.3 RAG 控制 Top K

RAG 不是检索越多越好。可以先从下面的设置开始：

- Top K：3 到 5。
- 每段 chunk 不要太长。
- 有 rerank 时，先召回多一点，再压到少量高质量片段。
- 把来源、标题、页码保留好，正文只放真正需要的部分。

### 7.4 控制输出长度

如果只是分类、抽取字段、判断是否需要工具，输出可以很短：

```json
{"intent": "search", "query": "向量库选择"}
```

不要让模型在每个中间步骤都写一大段解释。中间步骤短，最终回答再写完整。

### 7.5 工具结果要裁剪

工具返回值常常很长。不要把接口原始返回全部塞给模型。可以先在程序里做处理：

- 只保留关键字段。
- 只保留前几条结果。
- 去掉无关日志。
- 把大 JSON 转成短表格或短摘要。

## 8. 课堂里可以怎么观察

运行多轮对话脚本时，可以让程序打印三类内容：

```text
本轮输入消息数：5
估算输入字符数：3200
模型输出字符数：480
usage: {...}
```

如果平台返回 `usage`，重点看：

- 输入 token 是否随着轮次一直增长。
- 输出 token 是否被 `max_tokens` 控住。
- 是否出现 cached tokens。
- RAG 片段数量变化后，prompt tokens 是否明显变化。

这样学员能看到：多轮对话不是简单“继续聊”，背后其实是一次次重新组织上下文。

## 9. 一个实用判断

如果费用突然变高，先查四件事：

1. 是否把完整历史每轮都传回去了。
2. 是否把过多 RAG 片段塞进上下文。
3. 是否要求模型输出过长内容。
4. 固定内容是否没有放在前缀，导致缓存命中变差。

多轮对话的成本控制，本质上不是省掉必要信息，而是把“该让模型看的内容”整理得更短、更稳定、更靠前。
