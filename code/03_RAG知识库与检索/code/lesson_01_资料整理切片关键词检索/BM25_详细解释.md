# BM25 关键词检索详细解释

> 对应脚本：`04_关键词检索.py`、`05_检索结果解释.py`

BM25 是 RAG 系统里非常常用的关键词检索算法。它不理解语义，不会把“买票”和“创建订单”自动看成同义词，但它很擅长处理明确词、编号、术语、服务编号、条款号、服务点名称这类“必须精确命中”的内容。

在知识库里，BM25 经常和向量检索一起使用：BM25 负责抓住关键词，向量检索负责抓住语义相近的表达，最后再用 RRF 或重排模型把结果合并。

## 1. BM25 解决什么问题

最简单的关键词检索可以直接数命中次数：

```text
问题：候补申请一定能成功吗？

文档 A：候补申请不能保证成功……
文档 B：退款和变更需要查看订单状态……

如果问题里的“候补申请”“创建订单”“成功”在文档 A 中出现得更多，
文档 A 就应该排在前面。
```

但只数词频会有几个问题：

- 长文档天然更容易命中更多词，分数可能虚高。
- 一个词重复 20 次，不应该比重复 3 次重要 6 倍多。
- 常见词价值低，比如“什么”“需要”“可以”；稀有词价值高，比如“候补申请”“ORD-1001”“学生优惠票”。

BM25 就是在“词频命中”的基础上，加入了三类修正：

- 词频饱和：一个词命中越多越好，但收益会逐渐变小。
- 稀有词加权：越少见的词，命中后越重要。
- 文档长度归一化：长文档不会因为字数多就占便宜。

## 2. BM25 的公式

常见 BM25 公式如下：

```text
score(q, d) = Σ IDF(t) × ( TF(t, d) × (k1 + 1) )
                         / ( TF(t, d) + k1 × (1 - b + b × |d| / avgdl) )
```

其中：

| 符号 | 含义 |
| --- | --- |
| `q` | 用户问题，也就是 query |
| `d` | 一段文档或一个 chunk |
| `t` | 用户问题中的某个词 |
| `TF(t, d)` | 词 `t` 在文档 `d` 中出现的次数 |
| `IDF(t)` | 词 `t` 的稀有程度 |
| `|d|` | 当前文档长度 |
| `avgdl` | 全部文档的平均长度 |
| `k1` | 控制词频增长的饱和速度 |
| `b` | 控制文档长度归一化强度 |

课堂上不用一开始就背公式，可以先记住一句话：

```text
BM25 = 命中了哪些词 + 这些词有多稀有 + 文档长度是否合理
```

## 3. TF：词频不是越多越线性

TF 表示某个词在文档中出现多少次。

比如问题是：

```text
候补申请一定能成功吗？
```

分词后可能得到：

```text
候补申请 / 创建订单 / 一定 / 成功
```

如果某段资料里出现了：

```text
候补申请不能保证一定成功……
```

那么“候补申请”“创建订单”“成功”都命中了。

但 BM25 不会让词频无限线性增长。一个词出现 1 次有价值，出现 2 次更有价值，出现 10 次之后继续增加，价值提升就没那么明显了。

这叫“词频饱和”。

直观理解：

```text
出现 0 次：基本不相关
出现 1 次：有明显相关性
出现 3 次：更相关
出现 20 次：不应该比 3 次高很多，可能只是文档很长或重复啰嗦
```

## 4. IDF：越少见的词越重要

IDF 衡量一个词在整个资料库里有多稀有。

如果一个词在很多文档里都出现，它区分文档的能力就弱。

```text
“规则”这个词很多文档都有，区分度一般。
“候补申请”只在少数文档里出现，区分度更强。
“ORD-1001”这类服务编号号如果出现，往往非常关键。
```

常见 IDF 写法之一：

```text
IDF(t) = log((N + 1) / (DF(t) + 1)) + 1
```

其中：

| 符号 | 含义 |
| --- | --- |
| `N` | 文档总数 |
| `DF(t)` | 有多少篇文档包含词 `t` |

如果 `DF` 很小，说明这个词很少见，`IDF` 就高。

如果 `DF` 很大，说明这个词到处都有，`IDF` 就低。

## 5. 文档长度归一化

真实资料里，文档长度差异很大：

- FAQ 可能只有几十个字。
- 规章制度可能几千字。
- 操作手册可能几十页。

如果只看命中次数，长文档更容易“碰巧命中”很多词。

BM25 用这一项处理文档长度：

```text
1 - b + b × |d| / avgdl
```

它的意思是：

- 当前文档比平均文档长很多，分数会被适当压低。
- 当前文档长度接近平均值，分数比较自然。
- `b` 越大，长度惩罚越明显。

常用默认值：

```text
k1 = 1.5
b = 0.75
```

这两个参数在很多场景里已经够用。课堂和原型阶段可以先用默认值，不必一开始就调参。

## 6. k1 和 b 怎么理解

### k1：控制词频饱和

`k1` 越大，词频增长带来的收益越明显。

```text
k1 小：一个词出现几次后，继续重复的收益很快变小。
k1 大：重复出现更多次，分数仍会继续明显增加。
```

常用范围：

```text
k1 = 1.2 ~ 2.0
```

### b：控制长度惩罚

`b` 越大，文档长度对分数影响越大。

```text
b = 0：不考虑文档长度。
b = 1：完整考虑文档长度归一化。
b = 0.75：常用折中值。
```

如果你的知识库切片长度比较统一，`b` 的影响会小一些。

如果你的资料有的很短、有的很长，`b` 就更重要。

## 7. 和课程代码怎么对应

在 `04_关键词检索.py` 中，BM25 相关代码主要有三块。

### 1. 分词

```python
def tokenize(text):
    return [w for w in jieba.cut(text) if w.strip()]
```

中文检索通常要先分词。否则“候补申请”可能只被当成连续字符，无法稳定按词命中。

课堂上可以修改问题，例如：

```text
候补申请一定能成功吗？
退款费用怎么算？
学生优惠票需要什么条件？
```

观察分词结果和检索结果如何变化。

### 2. IDF 计算

```python
def idf(term, docs_list):
    df = sum(1 for d in docs_list if term in d)
    return math.log((N + 1) / (df + 1)) + 1
```

这段代码在算：

```text
某个词出现在多少篇文档里。
出现越少，IDF 越高。
```

### 3. BM25 得分

```python
numerator = tf * (k1 + 1)
denominator = tf + k1 * (1 - b + b * (doc_len / avgdl))
term_score = idf_val * numerator / denominator
```

这几行就是 BM25 的核心：

- `tf` 负责词频。
- `idf_val` 负责词的重要性。
- `doc_len / avgdl` 负责文档长度修正。
- `k1` 和 `b` 负责控制饱和与长度影响。

## 8. 一个小例子

假设问题是：

```text
候补申请一定能成功吗？
```

有三段资料：

```text
A：候补申请不能保证成功，兑现结果取决于退款、变更和新增库存。
B：办理退款款和变更前，应查看订单类型、服务开始时间和订单状态。
C：服务点A、服务点B、杭州东等服务点存在中文名、拼音和服务点编码。
```

BM25 大概率会把 A 排在第一，因为：

- A 同时命中“候补申请”“创建订单”“成功”。
- 这些词对问题很关键。
- A 的内容长度也比较合理。

B 可能排在后面，因为它命中了“退款”“变更”，和候补申请兑现有一点关系，但不是直接回答。

C 基本不相关，因为它主要讲服务点。

这就是 BM25 在关键词层面的价值：它能很快找到“字面上最相关”的资料。

## 9. BM25 适合什么场景

BM25 很适合：

- 服务编号号：`ORD-1001`、`D310`、`K123`
- 服务点名称：`服务点A`、`服务点B`
- 条款号：`第十七条`、`4.2.1`
- 产品名：`学生优惠票`、`候补申请`
- 固定术语：`退款费`、`变更`、`报销凭证`
- 日志编号、设备编号、工单编号

这些内容如果只靠向量检索，有时反而会被“语义相近”带偏。比如用户问 `ORD-1001`，你不能返回一个语义相近但服务编号不同的结果。

## 10. BM25 不擅长什么

BM25 的短板也很明显：

### 1. 不理解同义表达

```text
用户问：排队买票能不能保证？
资料写：候补申请不能保证成功。
```

如果没有命中“候补申请”，BM25 可能排不高。

### 2. 不理解上下文

BM25 看的是词命中和统计分数，不知道业务背景。

```text
“错过服务流程还能退吗？”
```

它可能需要依赖“错过”“退款”等词命中，但无法像向量模型那样理解完整语义。

### 3. 对分词质量敏感

中文分词如果切错，检索效果会明显下降。

例如：

```text
学生优惠票
```

如果被切成“学生 / 优惠 / 票”，有时可以；如果业务里需要整体术语，就应加入自定义词典。

## 11. 为什么要和向量检索混合

BM25 和向量检索各自擅长不同问题。

| 检索方式 | 擅长 | 不擅长 |
| --- | --- | --- |
| BM25 | 精确词、编号、条款号、术语 | 同义表达、语义改写 |
| 向量检索 | 语义相似、自然语言提问 | 精确编号、罕见词、强约束词 |

所以真实 RAG 系统里常用混合检索：

```text
用户问题
  ├─ BM25 关键词检索
  ├─ 向量检索
  └─ RRF / Rerank 融合排序
      └─ 送给大模型生成回答
```

本课程后面的 `lesson_03_混合检索与RRF` 会继续演示：

- BM25 先找关键词命中的资料。
- BGE-m3 向量检索找语义相近资料。
- RRF 把两路结果按排名融合。

## 12. BM25 在 RAG 里的位置

RAG 的基本链路可以写成：

```text
资料整理 → 文档切片 → 建索引 → 用户提问 → 检索 → 重排 → 生成回答
```

BM25 位于“建索引”和“检索”这两步：

```text
建索引阶段：
  - 对每个 chunk 分词
  - 统计词频 TF
  - 统计文档频率 DF
  - 保存倒排索引

检索阶段：
  - 对用户问题分词
  - 计算每个 chunk 的 BM25 分数
  - 返回 Top-K 片段
```

实际工程里，BM25 通常通过搜索引擎或检索库实现，比如 Elasticsearch、OpenSearch、Lucene、Whoosh 等。

课堂脚本为了让原理更透明，直接用 Python 手写了一个简化版。

## 13. 调参建议

### chunk size

BM25 和切片大小关系很大。

```text
chunk 太短：关键词可能分散在不同片段里，单个片段得分不高。
chunk 太长：容易命中很多词，但上下文噪声也更多。
```

常见做法：

- FAQ：一问一答作为一个 chunk。
- 规章：按条款或小节切。
- 手册：按标题层级、段落、表格说明切。

### overlap

overlap 对 BM25 也有帮助。

如果关键词刚好被切片边界拆开，BM25 可能检索不到完整信息。适当 overlap 可以减少这种问题。

### top_k

`top_k` 不是越大越好。

```text
top_k 太小：可能漏掉关键资料。
top_k 太大：会把无关片段带给大模型，增加噪声。
```

课堂原型可以先用：

```text
top_k = 3 或 5
```

真实项目需要结合评测集调。

## 14. 课堂讲解顺序建议

讲 BM25 时，可以按这个顺序：

1. 先从“数关键词命中次数”讲起。
2. 说明只数次数会偏向长文档。
3. 引出 IDF：稀有词更重要。
4. 引出词频饱和：重复命中不是线性增长。
5. 引出长度归一化：长文档不能天然占便宜。
6. 跑 `04_关键词检索.py`，观察 TF、TF-IDF、BM25 排序。
7. 跑 `05_检索结果解释.py`，看每个命中词对分数的贡献。
8. 最后衔接到混合检索：BM25 负责精确词，向量检索负责语义。

## 15. 常见问题

### BM25 是不是比向量检索落后？

不是。

BM25 是词法检索，向量检索是语义检索。它们解决的问题不同。很多生产级 RAG 系统仍然保留 BM25，因为精确词、编号、条款号非常重要。

### 有了大模型还需要 BM25 吗？

需要。

大模型负责组织回答，不负责保证资料一定被正确找到。RAG 的关键是先把资料找对。BM25 是“找资料”的重要方法之一。

### BM25 能不能直接生成答案？

不能。

BM25 只负责排序和召回。它返回的是相关文档或片段，不会自己写答案。生成答案需要大模型或模板。

### 为什么 BM25 分数没有固定上限？

BM25 分数不像余弦相似度那样通常落在 `-1 ~ 1` 或 `0 ~ 1`。它是多个词贡献的加和，所以不同查询之间的分数不能简单横向比较。

更常见的用法是：

```text
同一个查询下，比较不同文档的 BM25 分数排序。
```

## 16. 一句话总结

BM25 不是“智能理解文本”的算法，而是一个非常可靠的关键词排序方法。它通过词频、稀有度和文档长度修正，把“哪些资料字面上最相关”这件事做得很稳。在 RAG 系统中，它通常和向量检索配合使用，形成更稳的混合检索能力。
